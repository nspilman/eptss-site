What these lexicons describe

A community of people, on a recurring schedule, independently creating responses to a shared subject, then converging to share what they made.

EPTSS (cover songs) is the originating use case. The schemas are designed to fit the broader pattern: book clubs, recipe challenges, art prompts, writing circles, code katas, photography projects, film clubs.

Five records, dependency-ordered

  1. site.eptss.project
    the community
  2. site.eptss.subject
    the unit being responded to (song / book / recipe…)
  3. site.eptss.round
    one iteration with inline phases
  4. site.eptss.assignment
    prompts, instructions, or constraints attached to a round
  5. site.eptss.voteResult
    anonymous aggregate vote stats per (round, subject)

voteResult

site.eptss.voteResult

Anonymous aggregate vote stats for one subject in one round. Individual ballots are never published. Scale is self-describing per record.

FieldTypeReqNotes

subjectat-uriyes→ subject record

How records reference each other

All cross-references are at-uri strings (not CIDs) — stable across record updates; no cascading rewrites needed.

Design decisions baked in

  • Phases inline, not a separate record. ATProto has no joins — separate would force N+1 fetches in every consumer.
  • Subject IS a separate record. Has identity beyond rounds; re-covered songs share the same subject.
  • Vote scale is per-record (scaleMin / scaleMax). EPTSS is 1..5 today; lexicon doesn't lock that in.
  • Individual ballots never published — only aggregates.
  • Vote results gated on coveringBegins in the past. No in-flight tally publishing.
  • Communities emit native phase vocabulary. EPTSS uses signupOpens / coveringBegins / etc.; book clubs would use their own. No translation layer.
  • Phase semantics derivable from current time vs. phase events — no enum field that could go stale.
  • All cross-refs use at-uri strings (not CIDs). Stable across record updates.

Not in this first cut

  • User-attributed records (submission, reflection, comment, profile) — gated behind cutover at end of current round + per-user can_share_bsky consent.
  • DID ↔ Postgres user-id mapping column on users table. Needed for user-attributed records.
  • Decision: is ATProto canonical, or a mirror, for post-cutover user content? Shapes dual-write semantics.
  • Lexicon evolution / versioning strategy. Not needed yet; flagging.

Publish plan

  1. project stage — 2 records (cover, monthly-original)
  2. Spot-check on pdsls.dev
  3. subject stage — ~200 song records
  4. round stage — ~50 rounds, with inline phases + links
  5. assignment stage — round assignments
  6. voteResult stage — aggregates per (round, subject), finished rounds only
  7. Full re-run with no flags as audit pass — should report 0 failures
